# SKILL.md Authoring Template (self-contained)

Every `SKILL.md` in this stack **must** follow this exact section order and quality bar. This
template is local to the stack's skill-creator — it does **not** depend on any repo-global
`docs/` directory. The validator (`tools/validate.py` beside this file) enforces it and fails
the build on any deviation, placeholder phrase, or empty code fence.

> Golden rule: a skill exists to stop the agent from producing slop or bugs. If a section
> could apply to any stack unchanged, it is slop — make it specific, runnable, and verifiable.

---

## Required section order

```markdown
---
name: <stack>-<kebab-case-name>
description: <one concrete sentence; what the skill guarantees. No "Targeted domain-specific...">
related:
  - <stack>-<other-skill>
  - <stack>-<other-skill>
---

# <Title Case Name>
- 2 to 4 bullets stating the concrete outcome this skill enforces.

## Safety contract: non-negotiable
- Abort if <specific, detectable bad condition tied to a real failure mode>.
- Abort if <...>.

## Required tools
- <real runtime / package@version>.

## Gotchas
- <specific, version-aware trap that causes real bugs>.

## Workflow
1. <executable step>
2. <executable step>
3. <verification step>

## Code Examples (Good vs Bad)

### Bad Example 1 (<one-line caption naming the failure>)
```<lang>
real, runnable code that demonstrates the bug
```
### Bad Example 2..5 (<distinct failures>)
... five Bad examples total ...

### Good Example 1 (<one-line caption naming the fix>)
```<lang>
real, runnable, production-ready code
```
### Good Example 2..5 (<distinct correct patterns>)
... five Good examples total ...

## Related skills
- [[<stack>-<other-skill>]] — why these connect.
```

---

## Hard requirements (validator-enforced)

1. **Frontmatter**: `name` matches the directory name; `description` is specific and contains no
   banned phrase; `related` (if present) lists existing skill slugs in this stack.
2. **All six `##` sections present**, in the order above.
3. **≥5 Bad examples and ≥5 Good examples.** Every code fence has real code — no empty fences,
   no comment-only fences.
4. **`## Related skills`** has ≥1 `[[...]]` link, and every link resolves to a real skill dir
   in this stack.
5. **No banned placeholder phrases** anywhere in the file:
   - `Targeted domain-specific sub-skill`
   - `Verified secure execution paths`
   - `Draft implementation blueprint`
   - `Optimize configuration attributes`
   - `Run local unit tests assertions`
   - `Poin arsitektur tingkat tinggi`
   - `Autogenerated sub-skill`
   - `Missing boundaries verification`
6. **3-copy sync**: the file is byte-identical across `.agents/`, `.claude/`, and `.gemini/`
   (only `config.json` differs between agent dirs). Use `tools/sync.py` to mirror.

## Style notes

- Examples must be copy-pasteable and use the stack's real APIs and current idioms.
- Captions must name the *specific* failure, not "Bad code".
- Prefer linking a sibling skill over re-explaining it.
